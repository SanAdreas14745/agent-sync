# Структура Задачи Backlog Разработки

Этот reference описывает шаблон тела YouTrack-задачи для backlog разработки.

## Обязательный Шаблон

``` markdown
## User story

(Описание проблемы с точки зрения пользователя/бизнеса)

Я, как пользователь приложения, хочу ...

## Описание задачи

(Формальное описание)

Для того, чтобы пользователи могли ...

## Детали реализации

(Дополнительная информация для исполнителя, опционально)

- [ ] разработать сервис ...
- [ ] используя state ...
- [ ] в блоке модуля, начиная со строки ...

## Ожидаемый результат

(Информация, которая позволит исполнителю сравнить результат своей работы с тем, чего ожидает постановщик задачи)

Скриншот 1:


Комментарий для QA:

<чеклист для проверки>
```

## Разделы

### User story

Опиши пользовательскую или бизнес-проблему простым языком.

Формат по умолчанию:

``` text
Я, как <роль>, хочу <действие/возможность>, чтобы <ценность/результат>.
```

Если роль неизвестна, используй наиболее точную роль из контекста:
`пользователь`, `администратор`, `менеджер`, `оператор`, `клиент`,
`сотрудник поддержки`. Не придумывай роль, если она влияет на права доступа
или бизнес-логику: вынеси это в уточняющий вопрос.

### Описание задачи

Сформулируй, что должно измениться в продукте или системе.

Хорошее описание:

- содержит цель изменения;
- указывает затронутый сценарий;
- отделяет требуемое поведение от возможной реализации;
- не содержит внутренние рассуждения агента.

### Детали реализации

Используй раздел для информации, которая помогает исполнителю быстрее начать:

- найденные файлы, модули, компоненты, сервисы, state, API или контракты;
- предполагаемые точки изменения;
- важные ограничения совместимости;
- ссылки на существующие паттерны в проекте;
- короткие code blocks, если без них контекст легко потерять.

Пиши пункты как чеклист, когда они похожи на технические шаги. Если детали
реализации не нужны или могут преждевременно ограничить исполнителя, оставь
раздел с короткой фразой `Нет дополнительных технических ограничений.`.

### Ожидаемый результат

Опиши наблюдаемый результат, по которому постановщик и исполнитель смогут
сравнить реализацию с ожиданием.

Включай:

- что пользователь видит или может сделать после изменения;
- какие состояния или ошибки должны обрабатываться;
- какие данные сохраняются, отправляются или отображаются;
- что не должно сломаться;
- чеклист для QA.

Если пользователь приложил скриншоты, сохрани места под них:

``` markdown
Скриншот 1:

<описание или ссылка, если доступна>
```

Если скриншотов нет, не выдумывай их. Можно написать:
`Скриншоты не приложены.`

## Уточняющие Вопросы

Задай вопросы до составления финального тела, если неизвестно:

- какой пользовательский сценарий является целевым;
- какие роли или права доступа участвуют;
- какие данные, поля, статусы или интеграции должны измениться;
- какой проект, модуль или экран затронут;
- что считать успешным результатом для QA;
- нужно ли только подготовить текст или дождаться будущей MCP-интеграции для
  создания задачи.

Вопросы задавай отдельным сообщением. Не смешивай вопросы с готовым телом
задачи, если без ответов задача будет неточной.

## Интеграция С YouTrack

Создание задачи в YouTrack временно не реализовано в этом skill.

Используй заглушку:

``` text
Создание задачи в YouTrack будет реализовано позже через YouTrack MCP server.
Пока MCP-подключение не доступно, отправляю готовое тело задачи в чат.
```

Не проверяй локальные способы доступа и не пытайся создать задачу через прямой
API. До появления MCP-интеграции результатом работы skill является готовое
тело задачи.

## Критерии Качества

- Текст задачи можно вставить в YouTrack без дополнительной ручной переработки.
- Формулировки понятны человеку и достаточно структурны для агента.
- Каждый технический пункт опирается на запрос или найденные файлы.
- В задаче нет секретов, приватных значений конфигов и лишних логов.
- Acceptance criteria и QA checklist проверяемы вручную или автотестами.
- Не используются расплывчатые формулировки вроде `сделать нормально`,
  `поправить логику`, `улучшить экран` без конкретного ожидаемого поведения.
