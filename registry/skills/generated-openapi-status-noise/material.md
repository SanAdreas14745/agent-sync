# Generated OpenAPI Status Noise

## Назначение

Помогать отличать реальные изменения в autogenerated OpenAPI frontend files от
шума `git status` после перезаписи генератором, особенно на Windows.

## Главный Принцип

Не считай сотни файлов в `git status` содержательными изменениями, пока
`git diff --name-only` и `git diff --raw` не подтвердили реальный diff.

## Workflow

1. Ограничь проверку директорией generated OpenAPI files.
2. Сначала сравни `git status --short`, `git diff --name-only` и
   `git diff --raw` для этой директории.
3. Для лишнего файла из status проверь обычный diff, `git diff --quiet`,
   `git status --porcelain=v2` и EOL/attributes.
4. Если файл есть в status, но отсутствует в `git diff --name-only`, считай
   его несодержательным status noise или stat-dirty кандидатом.
5. Для stat-dirty файлов обнови index metadata через
   `git update-index --really-refresh`, не откатывая содержимое.
6. На Windows обновляй index чанками, если файлов много, чтобы не упереться в
   лимит длины аргументов.
7. После refresh повтори `git status --short` и `git diff --name-only`.
8. Реальные diff-файлы из чужих разделов не включай в задачу без явной
   необходимости; новые DTO/API по текущему Swagger/OpenAPI contract оставляй,
   если они относятся к задаче.

## Чек-Лист

- `status` отделён от содержательного `diff`.
- LF/CRLF рассматривается как гипотеза, а не как причина без подтверждения.
- Index metadata обновлена без отката содержимого.
- В задачу попали только релевантные реальные изменения.
