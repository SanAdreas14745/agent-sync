# Централизованное управление AI-скиллами для frontend-команды

Статус: planning-документ.

Этот документ фиксирует исходное предложение и эволюцию идеи AgentSync.
Финальную терминологию см. в [Glossary](../../docs/glossary.md), принятую
модель registry materials — в
[Registry Materials](../../docs/concepts/registry-materials.md).

## 1. Контекст

Frontend-команда работает с несколькими проектами:

-   около 10 проектов в разных репозиториях GitLab;
-   задачи распределены по разным пространствам и доскам YouTrack;
-   используется корпоративная база знаний Timly;
-   используются облачные сервисы Яндекса;
-   основная IDE команды --- WebStorm.

В процесс разработки активно внедряются AI-агенты, интегрированные в
IDE. Разные разработчики используют разные инструменты, например:

-   Codex;
-   Claude;
-   Cursor;
-   другие агенты, которые могут появиться в будущем.

На текущий момент правила, инструкции и скиллы для AI-агентов хранятся
локально у разработчиков. Например, инструкции для Codex могут
находиться в локальном файле `AGENTS.md` в каждом клоне проекта.

## 2. Проблема

Основная проблема --- отсутствие единого источника истины и
синхронизации AI-скиллов между членами команды.

Простой вариант с отдельным GitLab-репозиторием частично решает проблему
хранения, но создает дополнительные ограничения:

1.  Обновление скиллов через Merge Request является слишком медленным и
    неудобным для регулярного редактирования.
2.  Существуют разные уровни правил:
    -   общие для всей компании;
    -   общие для frontend-команды;
    -   проектные;
    -   специфичные для отдельного репозитория, если такой уровень
        действительно нужен;
    -   связанные с конкретным типом задачи.
3.  Разным AI-агентам требуются разные форматы инструкций. Один и тот же
    скилл нельзя гарантированно скопировать без изменений между Codex,
    Cursor, Claude и другими агентами.
4.  Требуется синхронизация изменений с локальными проектами
    разработчиков. Желательно, чтобы новый или измененный скилл
    автоматически становился доступен агенту после синхронизации или
    обновления проекта.

## 3. Предлагаемое решение

Создать внутреннюю платформу централизованного управления AI-скиллами.

Ключевая идея: скиллы хранятся в едином агент-независимом формате, а
затем автоматически преобразуются в формат, необходимый конкретному
AI-агенту.

Общая схема:

``` text
Web UI / Knowledge Base
          ↓
     Skill Registry
          ↓
 Skill Composer / Resolver
          ↓
 Agent Adapter Layer
          ↓
AGENTS.md / .cursor/rules / CLAUDE.md / другие форматы
          ↓
Локальный проект разработчика
```

## 4. Центральный реестр скиллов

Скилл должен быть самостоятельной сущностью, а не просто
Markdown-файлом.

Пример модели:

``` ts
type SkillSeverity = 'required' | 'recommended' | 'optional';

interface Skill {
  id: string;
  title: string;
  description: string;
  body: string;
  category: string;
  severity: SkillSeverity;

  scope:
    | 'company'
    | 'team'
    | 'project'
    | 'taskType';

  projects?: string[];
  technologies?: string[];
  paths?: string[];
  taskTypes?: string[];

  status:
    | 'draft'
    | 'active'
    | 'deprecated';

  version: number;
  author: string;
  updatedAt: string;
}
```

Скилл хранится в каноническом формате и не привязан напрямую к
`AGENTS.md`, `.cursor/rules` или другому конкретному формату.

В MVP публичный конфиг остается коротким: `project`, `agents`,
`technologies`, `registry`.

## 5. Иерархия правил

Предлагается разделить инструкции на несколько уровней:

``` text
Company
  ↓
Frontend Team
  ↓
Project
  ↓
Task Type
```

Примеры:

``` text
company.common
frontend.angular
frontend.code-review
project.statistics
task.api-contract
task.refactoring
```

При синхронизации система собирает итоговый набор инструкций для
конкретного разработчика и локального проекта:

``` text
1. Общие правила компании
2. Правила frontend-команды
3. Правила конкретного проекта
4. Дополнительные правила для выбранного типа задачи
```

Необходимо заранее определить правила приоритетов и разрешения
конфликтов между уровнями.

Например:

``` text
Company < Team < Project < Task Type
```

Более специфичное правило может переопределять более общее.

После пилота принято уточнение: `Repository` убран из публичной модели.
Различия frontend/backend и отдельных зон проекта выражаются через
`technologies`, `paths`, `category` и `taskType`.

## 6. Web-интерфейс

Для управления скиллами нужен человеко-ориентированный интерфейс, а не
обязательная работа с Git и Merge Request.

Пример структуры:

``` text
Скиллы
├── Общие
│   ├── Code Review Rules
│   ├── Angular Style Guide
│   ├── Testing Rules
│   └── YouTrack Workflow
│
├── Project A
│   ├── Architecture
│   ├── API Contracts
│   └── PrimeNG Conventions
│
└── Project B
    ├── Legacy Constraints
    └── Build and Deploy Rules
```

Основные возможности интерфейса:

-   просмотр скиллов;
-   создание;
-   редактирование;
-   удаление;
-   включение и отключение;
-   привязка к проектам, технологиям, путям и, при необходимости,
    репозиториям;
-   изменение области действия;
-   история версий;
-   просмотр diff;
-   rollback;
-   поиск;
-   фильтрация;
-   черновики;
-   публикация.

Для критичных глобальных правил можно предусмотреть отдельный процесс
согласования. Например, изменение обычного проектного скилла публикуется
сразу, а изменение company-level правила требует review.

## 7. Адаптация под разных AI-агентов

Между центральным хранилищем и локальным проектом нужен слой адаптеров.

``` text
Canonical Skill
├── Codex Adapter  → AGENTS.md
├── Cursor Adapter → .cursor/rules/*
├── Claude Adapter → CLAUDE.md
└── Generic Adapter → ai-rules.md
```

Разработчик создает правило один раз. Система отвечает за его
преобразование в формат конкретного агента.

Это позволяет:

-   не дублировать одни и те же правила;
-   централизованно обновлять инструкции;
-   подключать новые AI-агенты без миграции всей базы скиллов;
-   учитывать особенности конкретных агентов.

При этом нужно учитывать, что адаптер --- это не обязательно простое
копирование текста. Для разных агентов могут отличаться:

-   структура файлов;
-   метаданные;
-   область действия правила;
-   правила подключения файлов;
-   ограничения по размеру контекста;
-   механизм активации скиллов.

## 8. Skill Composer

Отдельный компонент должен определять, какие именно скиллы применимы к
конкретному локальному проекту.

Входные данные:

``` text
- пользователь;
- команда;
- Git remote;
- проект;
- технологии;
- пути изменяемых файлов;
- выбранный AI-агент;
- опционально — тип текущей задачи.
```

Результат:

``` text
Resolved Skill Set
```

Например:

``` text
frontend-common
+ angular
+ code-review
+ statistics-project
+ statistics-ui
= итоговый набор правил
```

Skill Composer также должен:

-   учитывать приоритеты;
-   обнаруживать конфликты;
-   исключать отключенные правила;
-   учитывать версии;
-   формировать детерминированный результат.

## 9. Локальный синхронизатор

На машине разработчика нужен небольшой CLI или локальный сервис.

Пример команд:

``` bash
ai-skills init
ai-skills sync
ai-skills watch
ai-skills status
ai-skills check
```

Пример процесса синхронизации:

``` text
1. Определить текущий Git remote.
2. Определить проект, технологии и опциональный task type.
3. Определить используемого AI-агента.
4. Запросить актуальный набор скиллов.
5. Собрать итоговую конфигурацию.
6. Применить адаптер конкретного агента.
7. Обновить локальные файлы.
8. Показать статус или diff.
```

Варианты запуска синхронизации:

-   при открытии проекта;
-   вручную;
-   раз в заданный интервал;
-   при запуске IDE;
-   через локальный watch-процесс;
-   после получения события от сервера;
-   перед началом новой AI-сессии.

Для первой версии достаточно ручной команды и синхронизации при открытии
проекта.

## 10. Конфигурация локального проекта

В Git проекта можно хранить только декларативную конфигурацию
подключения к платформе.

Например:

``` json
{
  "project": "statistics",
  "agents": ["codex", "cursor"],
  "technologies": ["typescript", "angular"],
  "registry": "../agent-sync-registry"
}
```

Этот файл:

-   хранится в репозитории;
-   определяет принадлежность проекта;
-   задает поддерживаемые агенты;
-   задает технологии и путь к registry.

Пути генерации файлов не нужно задавать в каждом проекте, если они
стандартны для адаптера. Для Codex adapter defaults:

``` text
AGENTS.md
.agents/generated/
```

Сами сгенерированные инструкции могут не храниться в Git и быть
добавлены в `.gitignore`.

## 13. Открытые вопросы для обсуждения с командой

Перед реализацией необходимо определить:

1.  Где физически хранить канонические скиллы:

    -   GitLab;
    -   отдельная база данных;
    -   Timly;
    -   Yandex Cloud;
    -   комбинированный вариант.

2.  Кто может:

    -   создавать скиллы;
    -   редактировать;
    -   публиковать;
    -   удалять;
    -   изменять глобальные правила.

3.  Нужен ли review для изменений.

4.  Какие AI-агенты необходимо поддержать в первой версии.

5.  Должны ли сгенерированные файлы храниться в Git.

6.  Как быстро изменения должны попадать на локальные машины.

7.  Нужна ли офлайн-работа.

8.  Как разрешать конфликты между общими и проектными правилами.

9.  Нужна ли привязка скиллов к:

    -   YouTrack-задаче;
    -   типу задачи;
    -   ветке;
    -   директории проекта;
    -   технологии.

10. Нужна ли аналитика:

    -   какие скиллы используются;
    -   какие версии установлены;
    -   какие агенты используются в команде.

## 14. Итоговая концепция

Предлагается создать единый источник истины для AI-инструкций команды.

Скиллы:

-   создаются и редактируются централизованно;
-   хранятся в агент-независимом формате;
-   распределяются по уровням и проектам;
-   автоматически собираются в итоговый набор;
-   адаптируются под конкретного AI-агента;
-   синхронизируются с локальными проектами разработчиков.

Таким образом, команда получает единый управляемый слой между
корпоративными знаниями и множеством AI-агентов, используемых
разработчиками.

## 15. Агент-ориентированная модель

При проектировании системы важно учитывать не только удобство
пользователей, но и то, как AI-агент реально получает и использует
инструкции.

Главный риск большой базы скиллов --- переполнение контекста и снижение
качества следования правилам. Если просто собирать все доступные
инструкции в один большой файл, агент может терять важные указания среди
второстепенных или тратить контекст на правила, не относящиеся к текущей
задаче.

Поэтому система должна быть не просто хранилищем Markdown-файлов, а
механизмом отбора минимального достаточного набора инструкций для
конкретного контекста.

### 15.0. Принятые уточнения после пилота

После проверки идеи на проекте `omega-1` зафиксированы следующие
решения:

-   `repository` убран из публичной модели и из `.ai-skills.json`. Для
    текущего сценария достаточно `project`, `technologies`, `paths` и
    `taskType`.
-   Пользовательский `output` убран из `.ai-skills.json`. Пути генерации
    задаются adapter defaults. Для Codex используются `AGENTS.md` и
    `.agents/generated/`.
-   Основной post-MVP сценарий распространения CLI --- глобальный
    npm-пакет `@ai-tools/ai-skills`. Registry скиллов планируется
    вынести в Yandex Object Storage, чтобы обновлять правила без
    перевыпуска npm-пакета.
-   Generated-файлы не должны быть одним большим документом. Агент
    получает короткий bootstrap, карту применимости и подробные
    category-файлы.
-   Для правил нужны поля `category` и `severity`, чтобы отделять
    truly required правила от рекомендательных и task-specific правил.

### 15.1. Формат скилла

Для первой версии рекомендуется хранить скиллы как Markdown-файлы со
строгим YAML frontmatter. Такой формат:

-   удобен для ручного редактирования;
-   хорошо отображается в Git diff;
-   может быть валидирован схемой;
-   легко переносится в Web UI в будущем;
-   позволяет отделить метаданные от текста инструкции.

Пример:

``` md
---
id: frontend.angular.forms
title: Angular Forms Rules
summary: Rules for working with Angular reactive forms
scope: team
category: frontend
severity: required
appliesTo:
  projects: ["statistics"]
  taskTypes: ["feature", "refactoring"]
  paths: ["src/app/**/*.ts"]
  technologies: ["angular"]
agents: ["codex", "cursor", "claude"]
priority: 300
loadMode: task
status: active
version: 4
owner: frontend
updatedAt: 2026-07-07
---

## Intent

When working with Angular forms in this project, prefer the established
reactive forms conventions.

## Rules

- Use reactive forms for complex forms.
- Do not mix template-driven and reactive forms in the same feature.
- Keep form validation rules close to form construction.

## Examples

...
```

Для быстрого поиска и резолвинга рядом может формироваться
скомпилированный индекс, например `registry.json`. CLI и Composer не
должны читать всю базу Markdown-файлов при каждом действии, если можно
использовать индекс.

Рекомендуемая модель метаданных:

``` ts
type SkillScope =
  | 'company'
  | 'team'
  | 'project'
  | 'directory'
  | 'taskType';

type LoadMode =
  | 'always'
  | 'project'
  | 'task'
  | 'onDemand'
  | 'reference';

interface Skill {
  id: string;
  title: string;
  summary: string;
  body: string;

  scope: SkillScope;
  category: string;
  severity: 'required' | 'recommended' | 'optional';

  appliesTo: {
    projects?: string[];
    paths?: string[];
    taskTypes?: string[];
    technologies?: string[];
    agents?: string[];
  };

  priority: number;
  loadMode: LoadMode;

  version: number;
  status: 'draft' | 'active' | 'deprecated';
  owner: string;
  updatedAt: string;
}
```

Ключевое поле для управления контекстом --- `loadMode`. Оно определяет,
как именно скилл должен попадать в итоговые инструкции агента.

### 15.2. Типы материалов

Не все материалы стоит называть скиллами. Рекомендуется разделить
сущности по назначению:

``` text
Rule
  Короткое обязательное указание.

Skill
  Поведенческий пакет для агента: как выполнять определенный тип работы.

Reference
  Длинный справочный материал, который агент читает только при
  необходимости.

Adapter
  Преобразователь канонических материалов в формат конкретного агента.
```

Примеры:

``` text
Rule:
  Не менять публичный API без обновления contract tests.

Skill:
  Как выполнять code review Angular-компонентов.

Reference:
  Архитектура проекта statistics-ui.
```

Такое разделение помогает не смешивать жесткие правила, рабочие
процедуры и справочную документацию.

### 15.3. Иерархия авторитета правил

Для разрешения приоритетов рекомендуется использовать иерархию:

``` text
Company
  ↓
Team
  ↓
Project
  ↓
Directory / Module
  ↓
Task Type
  ↓
Current User Request
```

Чем ниже уровень, тем конкретнее правило. При этом более конкретное
правило не должно молча отменять критичные company-level или safety
правила.

Рекомендуется явно разделить правила на категории:

``` text
non-overridable rules
  Обязательные правила, которые нельзя переопределить на нижних уровнях.

overridable defaults
  Общие настройки и рекомендации, которые могут уточняться проектом или
  конкретной областью проекта.

context-specific additions
  Дополнительные правила для конкретной директории, технологии или типа
  задачи.
```

### 15.4. Режимы загрузки в контекст агента

Для большой базы правил важнее не только иерархия владения, но и режим
загрузки в контекст агента.

Рекомендуемая модель:

``` text
Tier 0: Bootstrap
  Минимальные постоянные правила. Всегда попадают в главный файл.

Tier 1: Active Project Context
  Правила конкретного проекта, технологий и путей. Загружаются при sync.

Tier 2: Task Pack
  Правила под тип задачи: code-review, refactoring, API contract, tests.

Tier 3: On-demand Skills
  Не попадают в основной контекст. Агент видит только список или индекс и
  может запросить их при необходимости.

Tier 4: Reference Docs
  Большие справки, гайды, API agreements и архитектурные документы.
  Используются только по ссылке или через поиск.
```

Таким образом, агент получает короткий активный набор правил и карту
дополнительных знаний, а не всю базу целиком.

### 15.5. Структура generated-файлов

Для Codex и похожих агентов не рекомендуется генерировать один огромный
`AGENTS.md`. Лучше использовать короткий входной файл и отдельную
generated-директорию.

Пример:

``` text
AGENTS.md
.agents/
  generated/
    active-rules.md
    rule-scope.md
    skill-index.json
    rules/
      required.md
      frontend.md
      backend.md
      typescript.md
      html.md
      scss.md
      architecture.md
```

Пример содержимого `AGENTS.md`:

``` md
# Инструкции для агента

Этот репозиторий использует централизованные AI-правила.

Всегда начинай с:
- [.agents/generated/active-rules.md](.agents/generated/active-rules.md)

Карта применимости правил:
- [.agents/generated/rule-scope.md](.agents/generated/rule-scope.md)

Индекс доступных правил:
- [.agents/generated/skill-index.json](.agents/generated/skill-index.json)

Generated-файлы не редактируются вручную.
```

`active-rules.md` должен быть коротким bootstrap-файлом. В него попадают
только truly always-required правила: коммуникация, git safety,
контекст проекта, verification, dependencies и security.

`rule-scope.md` должен показывать агенту карту применимости: какие
category-файлы читать для `apps/front`, `apps/bff`, Swagger DTO,
операций с ветками и commit-сообщениями, code review и других
контекстов.

Подробные правила лежат в `.agents/generated/rules/*.md` и читаются
только когда соответствуют текущей задаче.

### 15.6. Context size diagnostics

Composer может показывать примерную оценку размера active context, но
пользовательский token budget в `.ai-skills.json` не входит в MVP.
Автоматическое обрезание правил недопустимо без отдельной политики:
нельзя терять critical/required rules только из-за грубой оценки размера.

### 15.7. Проверяемость и объяснимость

Чтобы агент и разработчик не упускали важные указания, система должна
быть объяснимой.

В каждый generated-файл рекомендуется добавлять manifest:

``` md
<!--
Generated by ai-skills
Included skills:
- company.common@3
- frontend.common@8
- project.statistics@5
Checksum: 4e2a...
-->
```

CLI должен поддерживать команды:

``` bash
ai-skills status
ai-skills check
ai-skills info frontend.angular.forms
```

Назначение команд:

-   `status` показывает активные скиллы и версии;
-   `check` находит конфликты, deprecated rules, слишком большие
    скиллы и дубли;
-   `info` объясняет, почему конкретный скилл попал или не попал в
    итоговый набор.

### 15.8. Рекомендации для MVP

С учетом агент-ориентированной модели MVP рекомендуется сузить до
проверяемого сценария:

``` text
Canonical Registry
  rules/
  skills/
  references/
  registry.json

Composer
  получает project + agent + technologies + taskType + path
  выбирает минимальный набор
  проверяет приоритеты, конфликты и бюджет

Adapter
  генерирует agent-specific структуру

Local Project
  AGENTS.md
  .agents/generated/*
```

Для первой версии достаточно:

1.  Markdown-файлов с YAML frontmatter;
2.  `.ai-skills.json` в локальном проекте;
3.  команды `ai-skills sync --agent codex`;
4.  генерации короткого `AGENTS.md` и
    `.agents/generated/active-rules.md`;
5.  генерации `.agents/generated/rule-scope.md` и
    `.agents/generated/rules/*.md`;
6.  поддержки `loadMode`: `always`, `project`, `task`, `reference`;
7.  полей `status`, `category`, `severity`, `priority`, `version`,
    `appliesTo`;
8.  команд `status`, `check`, `info`;
9.  без Web UI, БД и автоматического watch-процесса на первом этапе.

Главная ценность проекта в этой модели --- не только централизованное
хранение скиллов, а качественный Composer/Resolver, который формирует
для агента короткий, релевантный и проверяемый рабочий контекст.
