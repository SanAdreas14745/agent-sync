# Этап 0. Границы MVP

Статус: planning-документ.

Этот документ фиксирует границы и проверяемую гипотезу MVP.

## Цель

Зафиксировать минимальный объем первой рабочей версии, чтобы реализация
не расползалась в полноценную платформу до проверки главной гипотезы:
Composer / Resolver способен собрать для агента короткий, релевантный и
объяснимый набор правил.

## Проверяемая гипотеза

Если правила хранятся в файловом registry в виде Markdown + YAML
frontmatter, то CLI может выбрать нужные правила для локального проекта и
сгенерировать рабочие инструкции для Codex без базы данных, Web UI и
backend API.

## Сценарий MVP

``` text
Разработчик находится в локальном frontend-репозитории.
В репозитории есть .ai-skills.json.
Разработчик запускает ai-skills sync.
CLI читает файловый registry.
Composer выбирает применимые правила.
Codex adapter генерирует AGENTS.md и .agents/generated/active-rules.md.
Разработчик запускает Codex в проекте и агент видит актуальные правила.
```

## Входит в MVP

-   файловый registry;
-   Markdown-файлы с YAML frontmatter;
-   `.ai-skills.json` в локальном проекте;
-   один агент: Codex;
-   один основной output: `AGENTS.md`;
-   generated-директория `.agents/generated`;
-   Composer / Resolver как отдельный модуль;
-   Codex adapter как отдельный модуль;
-   CLI-команды:
    -   `sync`;
    -   `status`;
    -   `check`;
    -   `info`.

## Не входит в MVP

-   Web UI;
-   база данных;
-   backend API;
-   авторизация и роли;
-   review workflow;
-   Cursor adapter;
-   Claude adapter;
-   автоматическая фоновая синхронизация;
-   watch-процесс;
-   интеграция с WebStorm;
-   интеграция с YouTrack;
-   аналитика использования;
-   сложное разрешение смысловых конфликтов между правилами.

## Основные решения

### Источник данных

Первый источник данных --- локальная директория или отдельный
GitLab-репозиторий, подключенный как файловый registry.

База данных не используется в MVP, потому что нужно проверить не хранение
данных, а качество отбора и генерации инструкций.

### Agent support

В MVP поддерживается только Codex.

Это снижает количество переменных:

-   один формат выходных файлов;
-   один adapter;
-   проще проверить, как правила попадают в контекст агента;
-   проще собрать feedback от разработчиков.

### Generated-файлы

Generated-файлы считаются производными артефактами.

Рекомендуемая политика:

-   `.ai-skills.json` хранится в Git проекта;
-   `AGENTS.md` может храниться или не храниться в Git по решению команды;
-   `.agents/generated/*` лучше не хранить в Git;
-   все generated-файлы содержат предупреждение `Generated. Do not edit manually.`

Окончательное решение по Git-политике можно принять после пилота.

## Минимальные артефакты

В тестовом workspace должны появиться:

``` text
registry/
  skills/
    company/common.md
    frontend/common.md
    frontend/code-review.md
    projects/statistics/api.md

target-project/
  .ai-skills.json
```

После запуска `sync` в целевом проекте:

``` text
target-project/
  AGENTS.md
  .agents/
    generated/
      active-rules.md
      skill-index.json
```

## Критерии готовности этапа

-   сформулирован и согласован состав MVP;
-   понятно, какие компоненты будут реализованы первыми;
-   понятно, какие компоненты сознательно отложены;
-   есть пример структуры registry;
-   есть пример `.ai-skills.json`;
-   есть ожидаемая структура generated-файлов;
-   можно начать реализацию без дополнительных архитектурных решений.

## Риски

-   слишком рано начать проектировать платформу вместо CLI;
-   смешать требования будущего Web UI с требованиями MVP;
-   попытаться поддержать несколько агентов до проверки модели;
-   сделать слишком сложный формат правил;
-   не заложить объяснимость Composer-а с самого начала.

## Выход этапа

Документированная договоренность о границах MVP и наборе решений, на
которые опираются следующие этапы.
