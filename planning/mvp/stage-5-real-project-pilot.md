# Этап 5. Пилот на реальном проекте

Статус: planning-документ.

Этот документ фиксирует рабочий план пилота MVP на реальном проекте.

## Цель

Проверить MVP на одном реальном frontend-репозитории и понять, помогает
ли система агенту и разработчикам в обычной работе.

## Выбор проекта

Для пилота лучше выбрать проект, где:

-   есть активная разработка;
-   есть несколько типовых задач;
-   уже существуют локальные AI-инструкции или устные договоренности;
-   команда готова дать feedback;
-   нет критичного риска от generated-файлов.

Не стоит начинать с самого сложного legacy-проекта, если там много
исключений и неформальных правил.

## Подготовка

Нужно собрать начальный набор правил:

``` text
company.common
frontend.common
frontend.code-review
project.<project>.architecture
```

Для каждого правила нужно определить:

-   id;
-   scope;
-   loadMode;
-   appliesTo;
-   owner;
-   status;
-   version.

## Подключение проекта

В репозиторий проекта добавляется `.ai-skills.json`:

``` json
{
  "project": "statistics",
  "agents": ["codex"],
  "technologies": ["typescript"],
  "registry": "../agent-sync-registry"
}
```

Для Codex используются adapter defaults `AGENTS.md` и `.agents/generated/`.

В `.gitignore` можно добавить:

``` text
.agents/generated/
```

Решение по `AGENTS.md` нужно принять отдельно:

-   хранить в Git, если команда хочет единый entrypoint;
-   не хранить в Git, если все generated-файлы должны быть локальными.

## Проверочный сценарий

1.  Разработчик открывает проект.
2.  Запускает `ai-skills check`.
3.  Исправляет найденные проблемы, если они есть.
4.  Запускает `ai-skills sync`.
5.  Проверяет generated-файлы.
6.  Запускает Codex в проекте.
7.  Выполняет типовую задачу.
8.  Фиксирует, какие правила помогли, какие мешали, каких не хватило.

## Что измеряем

### Размер контекста

-   размер `AGENTS.md`;
-   размер `active-rules.md`;
-   estimated tokens;
-   количество included skills;
-   количество reference/on-demand skills.

### Качество выбора

-   все нужные правила попали в active-набор;
-   лишние правила не попали;
-   task-specific правила включаются только для нужного taskType;
-   reference материалы не перегружают активный контекст.

### Удобство поддержки

-   удобно ли писать frontmatter;
-   понятны ли ошибки валидации;
-   понятно ли пользоваться `check`;
-   понятно ли пользоваться `info`;
-   не слишком ли много ручной работы при добавлении правила.

### Поведение агента

-   агент видит ключевые правила;
-   агент не игнорирует critical rules;
-   агент не получает слишком много нерелевантного текста;
-   агент корректно ориентируется по generated-файлам.

## Feedback questions

Вопросы для разработчиков после пилота:

1.  Какие правила реально помогли?
2.  Какие правила агент проигнорировал?
3.  Какие правила оказались лишними?
4.  Какого правила не хватило?
5.  Удобно ли писать и редактировать скиллы?
6.  Понятно ли, почему правило попало в итоговый набор?
7.  Нужен ли `taskType` уже в повседневной работе?
8.  Нужно ли хранить `AGENTS.md` в Git?
9.  Нужен ли второй агент в следующей итерации?
10. Нужен ли Web UI или пока достаточно Git + CLI?

## Критерии успешного пилота

-   один реальный проект подключен через `.ai-skills.json`;
-   `ai-skills sync` стабильно генерирует инструкции;
-   разработчики могут понять active-набор через `status` и `info`;
-   `check` находит типовые проблемы;
-   Codex использует generated-инструкции;
-   команда видит пользу без ручного копирования правил;
-   список улучшений для следующей итерации понятен.

## Возможные решения после пилота

Если MVP полезен:

-   добавить Cursor adapter;
-   добавить Claude adapter;
-   добавить lock-файл версий;
-   добавить GitLab-hosted registry;
-   добавить review workflow для company-level правил;
-   добавить Web UI.

Если MVP перегружает контекст:

-   ужесточить `loadMode`;
-   сократить `always` правила;
-   перевести длинные материалы в `reference`;
-   добавить token budget enforcement;
-   улучшить summaries.

Если MVP неудобен в поддержке:

-   упростить frontmatter;
-   добавить генератор шаблонов;
-   улучшить `check`;
-   добавить auto-index;
-   добавить Web UI раньше.

## Выход этапа

Решение о следующей итерации на основе реального использования, а не
только архитектурных предположений.
