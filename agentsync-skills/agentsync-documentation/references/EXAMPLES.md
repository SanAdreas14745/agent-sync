# Примеры Ведения Документации

Этот reference показывает, как применять разделение `docs/` и `planning/`
в AgentSync.

## Финальная Концепция

Задача: описать принятое значение терминов `Rule`, `Skill`, `Reference`.

Правильное размещение:

``` text
docs/glossary.md
docs/concepts/<topic>.md
```

Почему:

- термины должны быть стабильным источником истины;
- читателю не нужны рассуждения о том, как команда пришла к определениям;
- финальный документ должен описывать текущее состояние, а не процесс выбора.

## Исследование Стороннего Проекта

Задача: изучить `Doctrine` и понять, какие практики можно перенять.

Правильное размещение:

``` text
planning/research/doctrine-skill-structure.md
```

Почему:

- это исследование, а не финальная модель AgentSync;
- выводы могут измениться;
- часть материала может позже быть переписана в `docs/`, но не вся
  исследовательская заметка должна стать финальной документацией.

## План Миграции Registry

Задача: описать шаги перехода от `registry/skills/*.md` к разделению
`rules/`, `skills/`, `references/`.

Правильное размещение:

``` text
planning/plans/registry-reorganization.md
```

Почему:

- это план работ;
- в нем допустимы этапы, риски, вопросы и временные решения;
- после выполнения итоговое состояние нужно описать отдельно в `docs/`.

## Плохой Вариант Для `docs/`

``` markdown
# Registry Materials

Пока думаем, что skill и rule надо разделить. Потом вернемся и решим,
как назвать папки. В Doctrine сделано вот так...
```

Проблемы:

- неясно, что уже принято;
- есть временная формулировка "потом вернемся";
- исследование Doctrine смешано с финальной моделью.

Такой материал должен быть в `planning/`.

## Хороший Вариант Для `docs/`

``` markdown
# Registry Materials

Registry material — каноническая единица хранения AgentSync.

Типы registry materials:

- `rule` — короткая норма поведения;
- `skill` — процедура выполнения задачи;
- `reference` — справочный материал, доступный по ссылке.
```

Почему это хорошо:

- текст описывает принятое состояние;
- нет временных рассуждений;
- термины сформулированы как источник истины.

## Promotion Пример

Было в `planning/research/doctrine-skill-structure.md`:

``` markdown
В Doctrine понравился подход, где `SKILL.md` не хранит все примеры.
Кажется, нам стоит попробовать `references/STRUCTURE.md` и
`references/EXAMPLES.md`.
```

После принятия решения в финальном документе:

``` markdown
Нетривиальный skill AgentSync должен хранить подробную структуру в
`references/STRUCTURE.md`, а примеры в `references/EXAMPLES.md`.
```

Что изменилось:

- убрано "кажется";
- убрано упоминание процесса исследования;
- сформулировано принятое правило.

## Типовые Ошибки

- Переносить planning-файл в `docs/` без переписывания.
- Хранить roadmap в `docs/`.
- Добавлять в glossary термин, который еще обсуждается.
- Оставлять в финальном документе фразы "потом поправить", "возможно",
  "пока считаем".
- Обновлять concept-документ, но забывать синхронизировать
  `docs/glossary.md`.
- Менять `README.md` во время внутренней уборки без явной необходимости.
